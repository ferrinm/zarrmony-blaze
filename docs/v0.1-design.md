# zarrmony-blaze v0.1 — implementation plan

## What it is

A zarrmony reader plugin that wraps **MACS iQ-processed Blaze exports** — the
per-`(channel, Z-plane)` OME-TIFF directories produced by the MACS iQ View
processing pipeline (Crop → Destripe → Denoise → Stitching → Contrast
Compression) — so `zarrmony convert <dir>` Just Works.

The plugin's promise is narrow: **read one processed Blaze experiment dir,
produce one OME-Zarr 0.5 store with full metadata.** No stitching, no raw-tile
input, no multiposition. Those land in v0.2 and v0.3.

## What zarrmony already provides — and what this plugin adds

Zarrmony's default reader routes `.ome.tif` to `bioio-ome-tiff` via `BioImage`.
On a real MACS iQ export, that path **fails** with `ParserError` because
`ome-types` (bioio-ome-tiff's XML parser) does a 2008→2016 schema upgrade and
then strictly rejects an `<AnnotationRef>` placement in the upgraded XML.

So this plugin earns its keep by:

1. Matching the *directory* (not a file path) so users don't have to know which
   internal file is the OME master.
2. Reading the OME-XML with stdlib `xml.etree.ElementTree` (no schema validation,
   no upgrade), sidestepping the `ome-types` failure.
3. Building a dask graph file-by-file from `<TiffData>` references rather than
   delegating to `tifffile.TiffFile.series`, which eagerly stats ~8k companion
   files at construction (unusably slow over network mounts).

Tradeoff captured in [ADR-0001](./adr/0001-tifffile-over-bioio-ome-tiff.md).

## Input contract

A directory containing files named:

```
<prefix>_Blaze_C<NN>_Z<NNNN>.ome.tif
```

- One file per `(channel, Z-plane)`.
- `_C00_Z0000.ome.tif` through `_C<N-1>_Z0000.ome.tif` are OME multi-file
  **masters** (`is_ome=True`); each carries the full XML for the whole series.
  The plugin reads the C0 master.
- All other files are `BinaryOnly` companions referenced from the master via
  `<TiffData>` UUID elements.
- No sidecar files. No companion `.ome.xml`. The whole metadata story is
  inside the master's `ImageDescription` tag.

## Reader Protocol contract (what the adapter exposes)

| Attribute | Value | Source |
| --- | --- | --- |
| `scenes` | `[<dir-basename>]` (always length 1 in v0.1) | The directory name |
| `set_scene(0)` | no-op | — |
| `xarray_dask_data` | `xarray.DataArray` with dims `(T, C, Z, Y, X)`, dask-backed, one chunk = one file = one `(Y, X)` plane | `<Pixels>` for shape; `<TiffData>` for the file map |
| `physical_pixel_sizes` | `(X, Y, Z)` in µm | `<Pixels PhysicalSizeX/Y/Z>` |
| `channel_names` | `list[str]` | `<Channel Name="...">` (or fluor/id fallback if Name missing) |
| `metadata` | the raw master OME-XML string | so zarrmony can drop `OME/source/raw.ome.xml` into the audit |
| `layout_hint` | `"flat"` | Blaze is light-sheet, not HCS |
| `close()` | implemented (no-op for stdlib XML, but forward-compatible) | — |

## Architecture

### Module layout

```
src/zarrmony_blaze/
├── __init__.py    # ReaderPlugin instance + _open() factory
├── match.py       # directory matcher
├── adapter.py     # BlazeReader class (Reader Protocol)
└── _ome_xml.py    # XML parsing — pure stdlib, fixture-testable in isolation
```

### Matcher

```python
def match(path: Path) -> int | None:
    if not path.is_dir():
        return None
    # Require both the _Blaze_ vendor token AND a Z0000 master file
    for entry in path.iterdir():
        n = entry.name
        if entry.is_file() and "_Blaze_" in n and n.endswith("_Z0000.ome.tif"):
            return 100
    return None
```

Side-effect-free. Single `iterdir()`, early return on first hit. The `_Blaze_`
literal prevents accidental hijack of unrelated OME multi-file directories.

### Open factory

```python
def _open(path: Path) -> BlazeReader:
    return BlazeReader(path)
```

### `_ome_xml.py` — the heart of the plugin

A pure function that takes the master XML string and returns a parsed
representation. No I/O. Testable against fixture XML strings.

```python
@dataclass(frozen=True)
class BlazeOme:
    size_t: int
    size_c: int
    size_z: int
    size_y: int
    size_x: int
    dtype: np.dtype                 # from PixelType (uint16 in observed data)
    pixel_size_x_um: float | None
    pixel_size_y_um: float | None
    pixel_size_z_um: float | None
    channel_names: list[str]        # length == size_c
    file_map: dict[tuple[int, int, int], str]  # (t, c, z) → filename (basename)

def parse_master_xml(xml: str) -> BlazeOme:
    ...
```

Notes for the implementer:
- Use `xml.etree.ElementTree.fromstring(xml)`. Namespace map:
  `{"ome": "http://www.openmicroscopy.org/Schemas/OME/2008-02"}`.
- `<Pixels>` is exactly one element. Read SizeT/C/Z/Y/X, PhysicalSizeX/Y/Z,
  PixelType (`uint16` maps to `np.uint16`; handle uint8/uint16/float32 at
  minimum).
- `<Channel>` elements live under `<Image>/<Pixels>`. Try `Name` attribute
  first, fall back to `Fluor`, fall back to `ID` (`"Channel:0"`, etc.).
- `<TiffData>` elements: missing `FirstT`/`FirstC`/`FirstZ` attributes default
  to 0. The `<UUID FileName="...">` child holds the companion filename
  (basename, not absolute).
- Validate after parsing: `len(file_map) == size_t * size_c * size_z`, all C/Z
  indices within bounds, no duplicates. Raise `BlazeMetadataError` otherwise.

### `BlazeReader` — Reader Protocol implementation

```python
class BlazeReader:
    layout_hint = "flat"

    def __init__(self, path: Path):
        self._dir = path
        masters = sorted(path.glob("*_C00_Z0000.ome.tif")) or \
                  sorted(path.glob("*_Z0000.ome.tif"))
        if not masters:
            raise BlazeMasterNotFoundError(...)
        master = masters[0]
        with tifffile.TiffFile(master) as t:
            xml = t.ome_metadata
        self._xml = xml
        self._ome = parse_master_xml(xml)
        self.scenes = [path.name]
        self._active = 0

    def set_scene(self, i: int) -> None:
        if i != 0:
            raise IndexError(...)
        self._active = 0

    @property
    def xarray_dask_data(self) -> xr.DataArray:
        # Build a dask array of shape (T, C, Z, Y, X), one chunk per (t,c,z)
        # plane, where each chunk lazily calls tifffile.imread(<file>).
        chunks = []
        for t in range(self._ome.size_t):
            t_chunks = []
            for c in range(self._ome.size_c):
                c_chunks = []
                for z in range(self._ome.size_z):
                    fname = self._ome.file_map[(t, c, z)]
                    fpath = self._dir / fname
                    delayed = dask.delayed(tifffile.imread)(str(fpath))
                    arr = da.from_delayed(delayed,
                                          shape=(self._ome.size_y, self._ome.size_x),
                                          dtype=self._ome.dtype)
                    c_chunks.append(arr[None, None, None, ...])  # add T,C,Z axes
                t_chunks.append(da.concatenate(c_chunks, axis=2))
            chunks.append(da.concatenate(t_chunks, axis=1))
        darr = da.concatenate(chunks, axis=0)
        return xr.DataArray(
            darr,
            dims=("T", "C", "Z", "Y", "X"),
            coords={"C": self._ome.channel_names},
        )

    @property
    def physical_pixel_sizes(self) -> PhysicalPixelSizes:
        o = self._ome
        return PhysicalPixelSizes(
            X=o.pixel_size_x_um, Y=o.pixel_size_y_um, Z=o.pixel_size_z_um
        )

    @property
    def channel_names(self) -> list[str]:
        return self._ome.channel_names

    @property
    def metadata(self) -> str:
        return self._xml

    def close(self) -> None:
        pass  # nothing to close; we hold no file handles between reads
```

The `from_delayed`-per-plane construction is naive but correct. If the dask
graph proves too large (8,380 nodes for a typical export — should be fine), the
fallback is `da.map_blocks` over a (T, C, Z) index grid, which keeps the graph
flat.

## Tests

Three layers, all small (≤200 LOC of test code total):

1. **`_ome_xml.py` unit tests** — feed in fixture XML strings (real master XML
   minus the bulk of TiffData), assert parsed fields.
2. **Matcher tests** — `tmp_path` fixtures with synthetic empty `.ome.tif`
   files; verify positive and negative matches, including the `_Blaze_`
   requirement.
3. **End-to-end conversion** — a `tmp_path` fixture with a tiny synthetic
   3-file series (1 channel × 3 Z-planes), real `tifffile.imwrite` calls to
   produce one master + two companions, run `zarrmony.convert`, assert on the
   output Zarr store. Mirrors the worked example in zarrmony's
   `writing-a-reader-plugin.md` §8.

## Known limitations (v0.1, by design)

- **One scene per directory.** Multiposition Blaze exports raise
  `NotImplementedError`. Workaround: convert one position at a time.
- **No raw-tile support.** v0.2 will add a separate matcher for raw dirs
  (`measurementInfo.txt` + `tiles.txt`) that emits **one scene per tile** with
  stage XY in `attrs.zarrmony.tile.*`. Stitching is delegated to BigStitcher,
  ASHLAR, m2stitch, or TeraStitcher.
- **`<AnnotationRef>` parsing not implemented.** We only extract the fields we
  need from the XML (channels, pixel sizes, file map). The full XML is
  preserved verbatim in the audit's `OME/source/raw.ome.xml`.

## Out of scope

- Stitching
- Reading raw tile output
- Multiposition / multi-acquisition
- Sibling raw dir auto-discovery (became moot when OME-XML was found in the
  master file — recorded in `CONTEXT.md`)
- Channel-name massaging beyond what's in `<Channel>` (vendor labels are used
  verbatim)
